# 核心设计信念

## 1. 权限先于展示

当前仓库把权限判断放在后端快照和前端守卫上，而不是让页面先渲染再禁用。这个约束值得保留，因为它减少了误导性的 UI 状态。

## 2. 后端是真相源

权限、tenant、会话、审批和通知的真相源都在后端。前端的职责是消费这些结果并把它们组织成清晰的工作流，而不是发明新的决策逻辑。

## 3. 工作台优先于品牌页

从 `Layout`、导航和 feature 结构来看，这个产品的重点是“完成受控工作”，不是“展示视觉品牌”。因此设计应优先强调：

- 找得到
- 做得完
- 留得下记录

## 4. 文档优先于抽象概念

知识库、文档审计、预览、下载、审批、归档是贯穿多个模块的主线。任何新的设计都应该先说明它如何服务这条文档主线。

## 5. 同源与简单部署优先

前端默认通过 Nginx 同源反代 `/api`，说明项目整体更偏向可控部署而不是复杂前后端分离架构。只要这个前提不变，设计上就不应过早引入额外的网络复杂度。

## 6. Fail-fast 比“温柔失败”更合适

仓库已有明确约束：缺少数据库、外部服务、路径或配置时应 fail fast。这一点同样适用于设计说明和产品说明，不应把未验证路径写成“系统会自动兼容”。

## 7. 真实约束必须写进文档

例如：

- `fronted` 目录名的历史包袱
- localStorage token 的安全代价
- `VALIDATION.md` 与当前工作区树结构的漂移

这些都不是“脏细节”，而是维护设计的一部分。
