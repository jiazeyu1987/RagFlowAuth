# 电子签名策略

版本: v1.1
更新时间: 2026-04-14
账号唯一性声明: 账号不得共用，不得代签
仓库外残余项: 线下离岗/转岗签名权限回收记录、账号唯一性培训签收记录、管理层批准版签名责任矩阵

## 1. 基本原则

- 电子签名仅代表签名人本人意图和责任。
- 电子签名必须绑定到当前步骤被授权审批人，不得由未被当前步骤授权的用户代签。
- 电子签名必须经过二次认证挑战并保留验签结果。
- 账号停权、禁用登录、离岗或转岗后，不得继续发起或完成电子签名。

## 2. 适用动作

- 审批通过
- 驳回
- 覆盖审批
- 质量域批记录签名与复核

## 3. 仓库内控制要求

- 每次签名必须记录 `signed_by`、`signed_by_username`、`signed_at_ms`、`action`、`meaning`、`reason` 和 `record_payload`。
- 签名令牌必须与签名用户一致；不同用户不得消费或复用他人的签名令牌。
- 账号当前不可登录时，系统必须阻断 `issue_challenge`、`consume_sign_token` 和最终签名落库。
- 当前步骤授权校验失败时，系统必须阻断签名并返回与当前审批步骤绑定的拒绝语义。

## 4. 审批授权绑定

- 授权来源: `approval_workflow_steps`
- 授权方式:
  - `approver_user_id`
  - `approver_role`
  - `approver_group_id`
  - `approver_department_id`
  - `approver_company_id`
- 当前步骤被授权审批人校验失败时，系统必须拒绝继续审批，不得生成签名记录。

## 5. 关键阻断点

- 未被当前步骤授权审批人发起签名时，必须阻断。
- 用户状态为停权、禁用或离岗时，必须阻断。
- 签名人与审批动作上下文不一致时，必须阻断。
- 电子签名事件必须进入统一审计日志并可验签。

## 6. 仓库外残余项

- 线下离岗/转岗签名权限回收记录
- 账号唯一性/禁共用声明培训签收记录
- QA 或管理层批准版责任矩阵

上述项目仍需在线下受控体系归档，仓库内不伪造为“已完成”。
