# 审批工作流 SOP

版本: v1.1
更新时间: 2026-04-14

## 1. 目的

规范受控文档多级审批流的配置、执行、变更和留痕要求，确保每个审批节点只由当前被授权审批人处理。

## 2. 适用范围

- 适用于 RagflowAuth 中所有进入审批流的受控文档和质量域对象。
- 适用于工作流配置、步骤审批、驳回和“待我审批”查询。

## 3. 角色职责

- 系统管理员：维护审批工作流模板，不直接替代业务审批。
- 审批人员：仅可处理自己被分配到的当前审批节点，并完成电子签名。
- 质量/合规负责人：复核工作流设计是否满足岗位、部门、公司边界要求。

## 4. 工作流配置规则

- 每条工作流至少包含 2 个审批步骤。
- 每个步骤必须至少配置一种审批绑定条件：`approver_user_id`、`approver_role`、`approver_group_id`、`approver_department_id`、`approver_company_id`。
- 步骤顺序必须连续，禁止跳号。

## 5. 审批执行要求

- 系统按当前步骤解析审批人。
- 非当前步骤被授权审批人不得执行通过或驳回。
- 非 `active`、已禁用登录或处于停权期的用户不得被视为当前步骤可签名审批人。
- 最后一步通过后对象才可转为 `approved`。

## 6. 留痕与复核

- 工作流配置变更应通过受控发布流程审批后生效。
- 每次审批/驳回都应保留文档标识、工作流与步骤信息、审批人标识、时间戳、电子签名和审计记录。

## 7. 例外处理

- 当前步骤未配置审批范围时，系统应拒绝保存工作流。
- 审批实例与工作流不一致时，系统应拒绝继续审批并返回错误。
