# 四类重构问题整理（2026-04）

## 文档定位

本文档用于整理以下四类已识别问题，并给出可执行的最小落地清单：

1. 职责边界混杂（同一模块承担过多角色）
2. 前后端规则双写/同步成本
3. 验证入口与当前文档树漂移
4. 安全基线已知缺口（默认密钥、token 存储策略）

执行约束：

- 不引入未被明确要求的 fallback、兼容分支或静默降级。
- 变更以“可验证、可回滚、最小影响面”为原则推进。
- 缺少前提时直接失败并报告缺失条件。

## 1. 职责边界混杂（同一模块承担过多角色）

### 问题定义

- 当前重构主计划已明确指出审批、通知、数据安全、文档预览等域存在“超级服务/超级 store/超长状态机”问题。

### 事实证据

- `docs/exec-plans/active/system-refactor-plan-2026-04.md`
  - 行 34-38：强调“优先拆分职责”“禁止新增隐式 fallback”
  - 行 82-83：审批域“超大服务 + 超大 store”
  - 行 138-141：数据安全 store 职责混杂与异常处理语义问题
  - 行 166-179：通知中心超级服务拆分目标
  - 行 238-251：文档浏览/预览状态机拆分目标
- `docs/exec-plans/active/system-refactor-plan-2026-04.md` 行 352-356 已将上述问题列入技术债沉淀建议。

### 影响

- 新需求进入热点模块时改动面大、回归链路长、故障定位成本高。

### 变更范围（最小）

1. 审批域：`service` 收敛为 facade，动作与仓储职责拆分。
2. 数据安全域：`store` 移除环境策略判断，只保留持久化职责。
3. 通知域：通道、规则、投递、inbox、目录同步拆为子服务。
4. 文档浏览/预览：将页面 Hook 与预览渲染按状态机拆分。

### 最小交付

1. 每个热点域至少形成 1 份“职责边界说明”（模块、输入、输出、事务边界）。
2. 每个热点域至少完成 1 个“高风险职责”拆分落地。
3. 为拆分出的新边界补齐 focused tests。

### 验收标准

- 关键入口文件从“承载领域细节”变为“编排/路由”角色。
- 对应域测试保持通过，新增行为有单测或集成测覆盖。
- 无新增隐式 fallback 分支。

## 2. 前后端规则双写/同步成本

### 问题定义

- 权限规则虽然已由后端快照驱动，但前端仍承载较多权限语义解释与包装函数，新增权限时仍有双端同步成本。

### 事实证据

- `QUALITY_SCORE.md` 行 18：新增资源类型需保证前后端同步扩展。
- `docs/exec-plans/active/system-refactor-plan-2026-04.md`
  - 行 201-202：目标是“后端成为唯一权限真相源”
  - 行 214-217：明确要清理前后端重复判断
- `fronted/src/hooks/useAuth.js` 行 202-250：前端存在大量 capability/permission 语义包装。
- `backend/app/core/permission_resolver.py` 行 170-212：后端构建权限快照。

### 影响

- 权限模型扩展时容易出现“后端已放开/前端未放开”或反向漂移。

### 变更范围（最小）

1. 固化后端 capability contract（resource/action/target）。
2. 前端 `useAuth.can()` 仅做薄适配，不继续扩散语义分支。
3. 将路由守卫、按钮显隐统一消费同一权限映射。

### 最小交付

1. 新增一份权限能力契约文档（输出字段、语义、示例）。
2. 下沉至少 1 组前端重复权限判断到统一适配层。
3. 增加“新增权限点”回归用例（前后端联动）。

### 验收标准

- 新增权限点时主要改动集中在后端 resolver + 前端一层映射。
- 同一权限语义不再散落多个前端文件重复实现。

## 3. 验证入口与当前文档树漂移

### 问题定义

- 当前仓库主文档树在 `docs/`，但验证契约仍以 `doc/e2e` 为核心，形成“双轨文档”事实；边界未被显式定义。

### 事实证据

- `AGENTS.md`
  - 行 16：强调不要把历史 `doc/` 结构当成当前事实
  - 行 32：声明 `docs/` 是当前主文档树
- `VALIDATION.md` 行 3、15、16：验证合同仍直接绑定 `doc/e2e`
- `doc/e2e/manifest.json` 行 3、30：实际仍在维护 `doc/e2e` 测试文档映射

### 影响

- 文档维护和验证执行的“权威来源”不清晰，容易导致维护分叉。

### 变更范围（最小）

0. 本阶段决策：保留双轨，明确边界（`docs/` 为主文档树，`doc/e2e` 为 E2E 业务脚本与 manifest 权威入口），不在 P1 做目录迁移。
1. 明确双轨边界：`docs/` 负责工程文档；`doc/e2e` 负责 E2E 业务脚本文档（如决定保留双轨）。
2. 或执行单轨迁移：将 `doc/e2e` 迁入 `docs/` 并一次性修正脚本入口。
3. 同步修正 `VALIDATION.md`、`AGENTS.md`、相关脚本帮助信息。

### 最小交付

1. 形成“文档权威来源说明”并落地到 `AGENTS.md` 和 `VALIDATION.md`。
2. 保证验证命令说明与真实目录结构一致。

### 验收标准

- 新成员按文档可一次性找到正确目录与正确验证入口。
- 不存在“文档写 A，脚本跑 B”的入口冲突。

## 4. 安全基线已知缺口（默认密钥、token 存储策略）

### 问题定义

- 后端存在可启动默认 JWT secret，前端 token 仍存放在 `localStorage`。

### 事实证据

- `backend/app/core/config.py` 行 25：`JWT_SECRET_KEY` 为示例默认值。
- `fronted/src/shared/auth/tokenStore.js` 行 13、22、35：access/refresh token 与 user 信息写入 `localStorage`。
- `docs/exec-plans/tech-debt-tracker.md` 行 20-21：以上两项已登记为开放技术债。

### 影响

- 生产环境若未覆盖默认密钥，身份校验强度不足。
- 浏览器端若发生 XSS，token 暴露风险上升。

### 变更范围（最小）

1. 后端增加启动期 fail-fast 校验：生产配置下禁止默认 JWT secret。
2. 前端形成 token 承载策略说明与迁移方案（短期约束 + 中期替代）。
3. 将安全基线校验纳入统一验证入口。

### 最小交付

1. 默认密钥校验落地并可由测试覆盖。
2. token 存储策略文档化，明确触发条件、风险和迁移步骤。

### 验收标准

- 生产环境配置不满足安全基线时，服务明确启动失败并给出原因。
- token 策略有明确 owner、时间表和可回归的检查项。
