# 新用户接入规格

## 1. 目标

定义一个新用户从“被系统录入”到“第一次进入并看到正确权限范围”的最小闭环。

## 2. 参与角色

- 管理员 `admin`
  负责创建用户、组织结构、权限组和必要配置
- 子管理员 `sub_admin`
  在被授权范围内管理知识库与部分配置
- 普通用户
  登录后消费后端权限快照，不自行决定访问范围

## 3. 前置条件

- `data/auth.db` 可读写
- 至少存在一个可分配的权限组
- 如需组织维度隔离，相关公司/部门已在系统中建立
- 如果用户需要访问知识库或工具，相关资源在后端已有真实可用配置

缺少任一条件时，应停止流程并显式修复，不应依赖默认成功或隐藏 fallback。

## 4. 推荐接入步骤

### 步骤 1：准备组织上下文

如果该用户需要公司、部门、直属管理者或 employee_user_id 信息，先在组织结构中准备好：

- 公司：`/api/org/companies`
- 部门：`/api/org/departments`
- 组织树：`/api/org/tree`

### 步骤 2：准备权限组

新用户不应只创建账号而不设计权限边界。至少需要确认：

- 分配哪些 `permission_groups`
- 是否需要可见知识库
- 是否需要可见工具
- 是否需要上传、审阅、下载、复制、删除等能力

如果用户没有有效权限组，后端快照通常会收敛到“默认拒绝”或空资源集合。

### 步骤 3：创建用户

管理员通过用户管理页或 `/api/users` 创建用户时，需要至少确定：

- `username`
- 初始密码
- `role`
- `status`
- `company_id` / `department_id`（如需要）
- `max_login_sessions`
- `idle_timeout_minutes`
- 是否允许修改密码
- 是否启用电子签名
- `managed_kb_root_node_id`（对子管理员尤其重要）

### 步骤 4：分配权限组

分配权限组后，后端在 `/api/auth/me` 中返回：

- `permission_groups`
- `permissions`
- `accessible_kbs`
- `accessible_kb_ids`
- `accessible_chats`

这是新用户第一次登录后真正看到页面和数据范围的依据。

### 步骤 5：首次登录

首次登录路径：

1. 前端提交 `/api/auth/login`
2. 成功后请求 `/api/auth/me`
3. `AuthProvider` 将快照写入本地存储
4. 再请求 `/api/me/kbs`
5. 路由壳根据 `role` 和 `permissions` 决定默认落地页与导航可见性

## 5. 首次登录后的预期

### 管理员

- 默认进入 `/logs`
- 可见管理导航
- 拥有全量权限快照

### 子管理员

- 默认进入 `/chat`
- 仅在可管理范围内看到知识库配置、权限组等入口
- 可能拥有受控的知识库目录根

### 普通用户

- 默认进入 `/chat`
- 看不到管理端入口
- 只能访问后端快照允许的知识库、聊天与工具

## 6. 失败条件

以下问题都应被视为 onboarding 失败，而不是“先让用户进来再说”：

- `/api/auth/me` 返回的权限快照与预期不符
- 用户第一次登录后看不到应有知识库
- 用户能看到未授权工具或管理入口
- `company_id` / tenant 作用域错误，导致请求命中错误数据库
- 必须依赖 RAGFlow 或 OnlyOffice 的能力未准备好

## 7. 验证建议

最小验证集合：

- 登录成功
- `/api/auth/me` 快照字段符合预期
- `/api/me/kbs` 返回符合预期
- 侧边栏仅显示应显示入口
- 默认落地页与角色匹配

如果验证失败，应先检查后端快照与权限组，而不是只改前端显示。
