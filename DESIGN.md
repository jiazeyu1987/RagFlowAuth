# 设计说明

## 1. 设计目标

这个项目的设计不是“展示品牌”，而是“让受控操作尽量少走弯路”。从代码结构看，当前设计目标更接近企业内部工作台：

- 权限优先：先决定能不能看、能不能做，再渲染页面
- 文档优先：知识库、上传、预览、审计、下载是主线
- 管理分层：普通用户、子管理员、管理员看到的导航和职责不同
- 同源优先：生产默认用 Nginx 反向代理 `/api`，降低跨域摩擦

## 2. 信息架构

当前信息架构大体分成五块：

- 个人工作区：聊天、搜索、我的密码、站内信
- 知识库工作区：上传、浏览、文档历史、知识库配置、搜索配置
- 管理工作区：用户、组织、权限组、通知设置、日志
- 审批与合规工作区：审批中心、审批配置、电子签名、培训合规
- 专用工具区：论文下载、专利下载、NAS 导入、药监导航、包装图纸

这说明产品不是围绕单一页面展开，而是围绕一组相互关联的受控业务能力展开。

## 3. 主要交互模式

### 3.1 路由即权限边界

`PermissionGuard` 说明一个核心设计决定：

- 未登录不允许进入业务路由
- 未授权也不先渲染再禁用，而是直接跳转到 `/unauthorized`

这让用户路径清晰，也减少了“页面可见但按钮不可用”的混乱状态。

### 3.2 布局驱动导航

`Layout` 把整个体验固定成：

- 左侧导航
- 顶部标题
- 主内容区
- 移动端抽屉侧栏

这意味着新页面通常只需要进入既有壳层，不应各自发明导航和页头模式。

### 3.3 页面 hook 负责复杂状态

大量页面采用 `useXxxPage.js` 作为控制器，例如知识库、文档浏览、审批配置、用户管理。这是一种“页面状态机下沉”的设计：

- 页面组件负责装配
- hook 负责加载、提交、错误和派生状态
- 局部组件负责具体渲染

对于当前仓库体量，这比把所有状态堆进单页组件更容易维护。

## 4. 关键设计原则

### 4.1 后端是权限真相源

前端 `useAuth.can()` 虽然做了二次封装，但真正的权限来源是后端 `auth/me` 返回的快照。设计上应继续维持这一点，不要把关键授权规则偷偷搬到前端。

### 4.2 文档与操作要能互相追踪

从文档上传、审计、下载、审批到电子签名，这个仓库明显强调“操作后可追踪”。设计上应优先保留：

- 操作日志
- 审批待办
- 电子签名记录
- 版本与归档信息

### 4.3 管理端和使用端要分层

`admin` 默认进入 `/logs`，普通用户默认进入 `/chat`，侧边栏还会对 admin 隐藏一部分普通用户导航。这体现了产品设计上的分层：管理员不是普通用户功能的超集，而是另一个工作入口。

## 5. 当前 UI 事实

从 `fronted/src/components/Layout.js` 和各 feature 组件看，当前 UI 特征是：

- 以内联样式和局部样式对象为主
- 没有独立 token 系统或组件规范文件
- 没有单独的设计系统依赖包
- 页面对话框、表格、列表、树结构广泛存在

因此当前设计文档应更多记录“已采用的模式”，而不是假装仓库已经拥有完善的设计系统。

## 6. 设计上的明显约束

- `fronted` 目录和现有脚本、Dockerfile 已深度绑定
- 预览能力涉及 PDF、OnlyOffice、Markdown、表格等多种文档形态
- 路由和权限配置高度耦合，改导航时必须同时看权限判断
- 专用工具区已经很大，后续设计最容易失控的地方就是“新工具继续往侧边栏堆”

## 7. 后续设计文档建议

当前最值得继续补写的设计主题是：

- 知识库信息架构和目录树设计
- 审批/签名/通知的交互闭环
- 专用工具入口的分组规则
- 移动端与桌面端的最小可用差异

详细入口见 [docs/design-docs/index.md](docs/design-docs/index.md)。
