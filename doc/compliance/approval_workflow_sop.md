# 审批工作流 SOP

版本: v1.1  
更新时间: 2026-04-02

## 1. 目的

规范受控文档多级审批流的配置、执行、变更和留痕要求，确保每个审批节点只由当前被授权审批人处理。

## 2. 适用范围

- 适用于 RagflowAuth 中所有 `pending` 状态受控文档。
- 适用于工作流配置、步骤审批、驳回和“待我审批”待办查询。

## 3. 角色职责

- 系统管理员：仅可维护审批工作流模板，不直接执行文档审批。
- 审批人员：仅可处理自己被分配到的当前审批节点，并完成电子签名。
- 质量/合规负责人：复核工作流设计是否满足岗位、部门、公司边界要求。

## 4. 工作流配置规则

- 每条工作流至少包含 2 个审批步骤。
- 每个步骤必须至少配置一种审批绑定条件：
  - 指定用户 `approver_user_id`
  - 指定角色 `approver_role`
  - 指定权限组 `approver_group_id`
  - 指定部门 `approver_department_id`
  - 指定公司 `approver_company_id`
- `approval_mode=all` 表示步骤内所有已配置条件必须同时满足。
- `approval_mode=any` 表示步骤内任一已配置条件满足即可。
- 步骤顺序必须连续，禁止跳号。

## 5. 审批执行要求

- 文档进入审批后，系统按当前步骤解析审批人。
- 非当前步骤被授权审批人不得执行通过或驳回。
- 非 `active`、已禁用登录或处于停权期的用户不得被视为当前步骤可签名审批人。
- 第 N 步通过后，实例推进到第 N+1 步；最后一步通过后文档才可转为 `approved`。
- 任一步驳回后，实例状态转为 `rejected`，并写入审批动作留痕。

## 6. 待我审批

- 前端默认展示“待我审批”视图，仅列出当前用户可处理的待审文档。
- 用户可切换到“全部待审”查看全量待审文档，但非当前步骤审批人不显示审批按钮。

## 7. 留痕与复核

- 工作流配置变更应通过受控发布流程审批后生效。
- 每次审批/驳回都应保留：
  - 文档标识
  - 工作流与步骤信息
  - 审批人标识
  - 时间戳
  - 电子签名与审计记录
  - 签名含义、签名原因和目标动作

## 8. 例外处理

- 若当前步骤未配置审批范围，系统应拒绝保存工作流。
- 若审批实例与工作流不一致，系统应拒绝继续审批并返回错误。
