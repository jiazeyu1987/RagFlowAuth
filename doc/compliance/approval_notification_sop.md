# 审批通知 SOP

版本: v1.0  
更新时间: 2026-04-02

## 1. 目的

规范多级审批流中的待审批通知、结果通知、失败重试、人工补发和升级通知处理，确保通知对象与当前审批节点保持一致，并可被审计追溯。

## 2. 适用范围

- 适用于 RagflowAuth 中受控文档的审批待办通知。
- 适用于邮件和钉钉两类通知通道。
- 适用于管理员对通知任务的查询、重试和人工补发。

## 3. 收件人规则

- `review_todo_approval` 仅发送给当前审批节点解析出的被授权审批人。
- 节点收件人按工作流步骤字段实时解析：
  - `approver_user_id`
  - `approver_role`
  - `approver_group_id`
  - `approver_department_id`
  - `approver_company_id`
  - `approval_mode`
- 第 1 步只通知第 1 步审批人；第 2 步只通知第 2 步审批人；非当前审批人不得收到待审批通知。
- `review_approved` 与 `review_rejected` 发送给文档上传人，用于结果回执。

## 4. 通道地址来源

- 邮件地址来自 `users.email`。
- 钉钉接收人标识来自通知通道配置 `recipient_map`：
  - key 优先使用 `user_id`
  - 若未命中，再使用 `username`
  - value 为钉钉 `userId`
- 若当前事件在所有启用通道上都无法解析出可投递地址，则该通知任务不得伪造成功，系统应记录失败原因。

## 5. 通知模板最小字段

所有审批通知 payload 至少包含：

- `current_step_name`
- `filename`
- `doc_id`
- `workflow_id`
- `approval_target.doc_id`
- `approval_target.step_no`
- `approval_target.step_name`
- `approval_target.route_path`

模板规则：

- 待审批通知标题必须体现“待审批 + 文档名 + 步骤名”。
- 结果通知标题必须体现“已批准”或“已驳回 + 文档名”。
- 正文必须包含审批定位字段，至少能通过 `route_path` 或 `doc_id + workflow_id + step_no` 回到审批目标。

## 6. 去重与重复抑制

- 同一事件、同一文档、同一步骤、同一收件人、同一通道使用统一 `dedupe_key`。
- 当存在同 `dedupe_key` 的既有任务时，系统不得重复创建新的待审批通知任务。
- 人工补发必须创建新的通知任务，并记录 `source_job_id` 指向原任务。

## 7. 重试、补发与升级通知

自动重试：

- 投递失败的任务按任务配置 `max_attempts` 自动重试。
- 每次失败都必须记录错误信息和尝试时间。

人工补发：

- 管理员在 `/api/admin/notifications/jobs/{job_id}/resend` 触发补发。
- 补发任务必须保留原 payload、原收件人，并生成新的 `dedupe_key` 和新的任务记录。

升级通知：

- 若同一任务达到最大重试次数仍失败，由管理员查询失败日志后人工判定升级路径。
- 升级通知不得替代原任务记录，必须作为受控的人工处置动作单独留痕。
- 升级方式可选择：
  - 修复原通道后执行人工补发
  - 启用备用已验证通道后重新补发
  - 通知质量/合规负责人执行人工跟催，并在偏差记录中登记

## 8. 审计与复核

- 通知任务必须保留：
  - 通道
  - 事件类型
  - 收件人用户
  - 收件地址
  - payload
  - 去重键
  - 源任务编号（补发时）
  - 投递状态
  - 重试次数
  - 错误原因
- 管理员每次版本发布后应抽查：
  - 首步待审批通知是否只发给当前节点审批人
  - 步骤推进后待审批通知是否切换到下一步审批人
  - 结果通知是否返回上传人
  - 重复触发是否被抑制
  - 补发任务是否可追溯到原任务
