# 电子签名授权与验签

## 全真链路状态

- 自动化状态：已接入 `doc/e2e` 全真链路
- 对应自动化：`fronted/e2e/tests/docs.electronic-signatures.spec.js`
- 运行入口：`python scripts\run_doc_e2e.py --repo-root .`
- 真实数据来源：`scripts/bootstrap_doc_test_env.py` 中真实签名记录与真实授权用户
- 当前已覆盖：签名详情查看、验签、授权停用、授权恢复
- 当前未覆盖：业务端在“无签名资格”下的拒绝路径、更多签名动作类型

## 流程目标

验证主管理员启用或停用签名资格后，审批或审核角色的签名动作是否随之变化，并验证签名结果能否被管理端追踪和验签。

## 参与角色

- 主管理员
- 普通用户审核员或审批人

## 前置条件

- 系统已启用电子签名能力。
- 审核员或审批人存在需要签名确认的业务动作。
- 主管理员能够访问电子签名管理。
- 开始测试前先确认目标用户当前的签名资格状态，避免把默认状态误判为“尚未授权”。
- 自动化执行时验签与授权切换必须走真实后端，不得用假签名结果替代。

## 场景一：签名资格启用时完成签名审批

### 测试步骤

1. 主管理员进入电子签名管理，确认目标审核员或审批人的签名资格为启用状态；如当前为停用，则先启用。
2. 审核员或审批人登录系统，执行需要签名的通过或驳回动作。
3. 主管理员进入电子签名管理查看新生成的签名记录。
4. 主管理员对该记录执行验签。

### 预期效果

- 在签名资格启用时，目标角色能够完成带签名的审批或审核动作。
- 签名动作完成后，系统会生成签名记录。
- 主管理员可以查看签名主体、签名时间、签名含义和签名结果。
- 验签后，系统应明确给出验签通过或失败结论。

## 场景二：停用签名资格后再次验证

### 测试步骤

1. 主管理员停用目标角色的签名资格。
2. 让该角色再次执行需要签名的业务动作。

### 预期效果

- 停用签名资格后，该角色不应再按原方式完成签名业务。
- 系统应阻止其继续执行依赖签名资格的关键动作，或明确提示当前无签名资格。
- 管理端不应继续生成不合法的新签名记录。
